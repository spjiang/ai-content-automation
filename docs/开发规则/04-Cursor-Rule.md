# Cursor Rule

## 1. 什么是 Cursor Rule

`Cursor Rule` 可以理解为“仓库级 AI 行为约束”。  
它的作用不是定义某次变更要做什么，也不是定义 AI 的通用工作流，而是告诉 AI：

- 在这个项目里必须遵守哪些长期规则
- 哪些技术选型已经确定
- 哪些安全与协作边界不能突破

简单说，`Cursor Rule` 更像当前仓库的“AI 执行制度”。

## 2. 当前项目里有哪些 Cursor Rule

从仓库内容看，当前项目至少有这些规则文件：

- `.cursor/rules/core-ai-standards.mdc`
- `.cursor/rules/git-collaboration.mdc`
- `.cursor/rules/security-config.mdc`

这些文件不是普通说明文档，而是会在 AI 协作中持续生效的项目规则。

## 3. 它们在当前项目里分别管什么

### 3.1 `core-ai-standards.mdc`

这是总规则，覆盖了：

- 工作流纪律
- 后端技术规范
- 前端技术规范
- Docker 与环境规范
- 测试与质量门禁

例如它明确要求：

- 需求澄清 -> 方案设计 -> 实施 -> 验证 -> 总结
- 涉及需求或范围变化时，先更新 OpenSpec 工件
- FastAPI 路由统一在 `/api/v1`
- Schema 变更仅通过 `Alembic`
- 长耗时任务必须走 `Celery + RabbitMQ`

这类规则直接决定 AI 和开发者在仓库里的默认做事方式。

### 3.2 `git-collaboration.mdc`

这个规则文件更偏协作与提交质量，强调：

- 新功能和修复应走分支开发
- 提交要小步、可回滚
- Commit message 使用规范前缀
- 提交前尽量通过对应模块的 lint 与测试

它的价值在于让“AI 参与开发”不破坏团队协作秩序。

### 3.3 `security-config.mdc`

这个文件主要负责安全和配置边界，例如：

- 禁止提交密钥、令牌、连接串
- 凭证通过环境变量注入
- 日志中不能输出敏感字段
- 外部 API 要有超时、重试、降级策略

这部分规则非常关键，因为 AI 在高效率生成内容时，也可能高效率制造安全风险。

## 4. 它和 OpenSpec 的区别

这是最容易混淆的地方之一。

- `OpenSpec`：描述某一次变更要做什么
- `Cursor Rule`：描述无论做哪次变更，都必须遵守什么

比如：

- “本次要新增审核 API 和发布包导出”属于 `OpenSpec`
- “所有 API 统一放在 `/api/v1`”属于 `Cursor Rule`

一个是“变更内容”，一个是“长期制度”。

## 5. 它和 Superpower 的区别

- `Superpower` 更偏 AI 的工作流
- `Cursor Rule` 更偏仓库长期约束

例如：

- “先做设计再实施”更接近流程方法
- “禁止硬编码密钥”是硬性边界

所以 `Superpower` 规定“做事顺序”，`Cursor Rule` 规定“做事底线”。

## 6. 当前项目里为什么需要 Cursor Rule

这个项目的技术栈、运行环境和安全边界已经很明确，如果没有规则固化，AI 或开发者都可能产生偏差：

- 可能绕开 `OpenSpec` 直接编码
- 可能用不符合仓库规范的技术方案
- 可能忽略 Celery、Alembic、Docker Compose 这些既定要求
- 可能无意间引入安全问题

`Cursor Rule` 的本质，就是把这些团队共识固化成执行护栏。

## 7. 对开发者的价值

开发者使用 AI 时，最怕的是 AI 给出“看起来能跑，但不适合当前仓库”的方案。  
`Cursor Rule` 可以显著减少这种偏差。

开发者重点要看的是：

- 当前仓库默认技术栈是什么
- 哪些实现方式是强制要求
- 哪些行为是禁止的
- 哪些质量门禁在提交前必须考虑

这样你就不会把通用建议误当成项目最佳实践。

## 8. 对管理者的价值

管理者更应该把 `Cursor Rule` 看成治理工具：

- 把关键技术约束写死，减少口头传达损耗
- 把安全底线和质量门禁前置
- 把协作规则交给系统长期提醒
- 让 AI 使用方式更加可管控

对于管理者而言，它的最大价值不是“解释技术”，而是“固化组织规则”。

## 9. 当前仓库里最值得重点关注的规则

### 9.1 先规格后编码

当前规则明确指出：

- 涉及需求或范围变更时，先更新 OpenSpec 工件，再编码

这说明本项目不接受“边写边想”的随意式开发。

### 9.2 技术栈不可随意漂移

规则已经明确：

- 后端以 `Python 3.11 + FastAPI` 为基线
- 数据持久化以 `PostgreSQL` 为准
- Schema 变更只能走 `Alembic`
- 长耗时任务必须走 `Celery + RabbitMQ`

这对开发者和 AI 都是很强的限制条件。

### 9.3 安全配置必须外置

规则明确禁止把密钥、连接串、令牌硬编码进代码或配置。  
这和当前仓库中存在未跟踪 `.env` 文件的现实情况尤其相关，更需要保持敏感信息不入库。

## 10. 适用场景

`Cursor Rule` 适合所有需要长期维护、多人协作、AI 频繁参与的项目。  
尤其适合：

- 技术栈固定的项目
- 安全与配置要求严格的项目
- 希望 AI 不偏航的团队
- 希望项目规范能长期沉淀的组织

## 11. 常见误区

### 误区 1：把 Rule 当成普通文档

不是。  
普通文档只是给人读，`Cursor Rule` 还会影响 AI 的执行行为。

### 误区 2：觉得 Rule 越少越灵活越好

短期可能更自由，但长期会导致输出不一致、质量不稳定、治理成本更高。

### 误区 3：以为 Rule 只对 AI 有用

其实它对人也同样有价值。  
因为它把项目共识写成了可持续复用的规则资产。

## 12. 一句话记忆

在当前项目里，`Cursor Rule` 的定位就是：  
把团队对技术栈、安全、流程和协作的长期要求，固化为 AI 和开发者都必须遵守的仓库规则。
